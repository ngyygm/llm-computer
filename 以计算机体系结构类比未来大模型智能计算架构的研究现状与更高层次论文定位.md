# 以计算机体系结构类比未来大模型智能计算架构的研究现状与更高层次论文定位

## 核心判断

有，而且已经形成了相当清晰的“近邻文献群”，只是这些工作大多各自占住体系结构中的某一层，还没有完全收敛成一篇被广泛接受的、全栈意义上的“未来智能计算体系结构”总论。与你想法最接近、且学术上最值得正面回应的主线，至少包括五类：把 LLM 视为操作系统或内核的工作；把 agent 系统按 von Neumann 或 stored-program computer 重写的工作；把上下文与长期记忆按内存层级管理的工作；把工具与外部系统视为 I/O 的工作；以及把多智能体系统视为互联网络或分布式系统的工作。最关键的几篇拼图是 Ge 等的 *LLM as OS, Agents as Apps*、Mei 等的 *AIOS*、Packer 等的 *MemGPT*、Holt 等的 *L2MAC*，以及 Mi 等从 computer systems 视角撰写的综述。citeturn16view0turn15view0turn17view0turn20view0turn14view0

如果把你最核心的一句“LLM 很像 CPU”单独拎出来看，文献里也已经出现了相当接近的直接表述，但它在主流文献中的成熟度，反而弱于 “LLM as OS/kernel” 这条线。到 2025 年，ArbiterOS 已经明确把 LLM 称为“probabilistic processors”，并进一步发展出“Probabilistic CPU + deterministic governor/kernel”的分层；到 2026 年，还有一篇非主流平台 aiXiv 上的预印本直接以 *Treating LLM as the CPU* 为题。换句话说，**你的直觉并不是空白地带**；真正的空间不在于第一次说出这个类比，而在于把这些零散类比提升成一套更自洽的、可比较的、可评价的全栈架构理论。citeturn25view0turn13search0turn2search0turn2search1

更重要的是，现有工作之间其实存在一个尚未被彻底解决的分歧：模型到底更像 CPU，还是更像 OS kernel，还是更像一种“reasoning kernel”？Ge 等把 LLM 放在操作系统内核的位置；AIOS 将 LLM 与资源管理、上下文管理、访问控制一起纳入内核式服务；AgentOS 则把 LLM 视为由 OS logic 约束的 “Reasoning Kernel”；而 ArbiterOS 又把 LLM 下沉为受治理的“Probabilistic CPU”。这说明文献共同感受到“传统系统抽象正在迁移到 agent/LLM 栈”，但还没有形成统一的层次边界。这个“边界不清”本身，就是你论文可以抬高一个层次的入口。citeturn16view0turn15view0turn21view2turn25view0

## 与你设想最接近的文献谱系

第一条最直接的谱系，是 **LLM-as-OS / agent runtime / kernel**。Ge 等在 2023 年就系统地提出：LLM 对应 OS kernel，context window 对应 memory，external storage 对应 file system，hardware tools 对应 peripheral devices，software tools 对应 programming libraries，user prompts 对应 user commands；他们甚至把“Agents as Apps”“Natural Language as Programming Interface”一起放进了同一张系统图里。Mei 等随后在 *AIOS* 中把这一思路推进到可运行的 agent OS：将调度、上下文管理、记忆管理、存储管理、访问控制从 agent application 中剥离出来，放进 AIOS kernel，并报告最高 2.1 倍的运行加速。到 2025—2026 年，这条线继续向桌面和移动端推进，UFO2 把 HostAgent、AppAgent、GUI–API action layer 和虚拟桌面并行操作组织成 “Desktop AgentOS”，Aura 则进一步把 System Agent、sandboxed App Agents 和 Agent Kernel 做成安全优先的移动 agent OS。citeturn16view0turn15view0turn21view0turn22view0

第二条与你非常贴近的谱系，是 **von Neumann / stored-program computer / automatic computer**。Holt 等在 ICLR 2024 的 *L2MAC* 中，明确把其框架称为“the first practical LLM-based general-purpose stored-program automatic computer (von Neumann architecture) framework”，并给出 instruction registry、file store、control unit、separate LLM agents 的结构化设计。Mi 等在 2025 年进一步从“computer systems perspective”组织 LLM agent 综述，明确表示其框架受 von Neumann architecture 启发，并把 agent 拆成 perception、cognition、memory、tool、action 等模块，同时指出 context window 类似主存、数据库类似磁盘，而 agent 目前缺的正是“cache module”。这两篇文献说明：**“用计算机体系结构视角重述 agent/LLM 系统”本身已经是存在的研究命题**，且最学术化、最适合作为你论文正面回应对象的，正是这一支。citeturn20view0turn14view0

第三条是 **记忆层级与“操作系统式记忆管理”**。MemGPT 的核心不是简单加一个 RAG，而是借用传统操作系统中的 hierarchical memory systems 和 virtual context management，把有限 context window 扩展成一种“看上去更大”的工作上下文，其控制流还显式使用 interrupts。MemOS 则把问题抬到更系统的层次：把 plaintext、activation-based、parameter-level memories 统一成可表示、可调度、可演化的系统资源，用 MemCube 封装内容与 provenance/versioning，并试图在 retrieval 与 parameter learning 之间建立迁移桥梁。你的“KV 缓存像高速缓存、上下文管理像内存管理”的想法，与这一条线高度同构，但需要注意，最强的现有文献已经不再满足于“类比”，而是开始讨论 memory lifecycle、scheduling 和 evolvability。citeturn17view0turn18view0

第四条是 **工具、I/O 与总线协议**。在方法层面，ReAct 将 reasoning 与 acting 交织起来，把外部 API 或环境交互拉进推理回路；Toolformer 进一步让模型学会何时调用 API、传什么参数、如何合并结果；HuggingGPT 把 LLM 放到“controller”位置来调度 Hugging Face 上的异构模型；Gorilla 则聚焦 API call 的准确性与文档更新适应性。到标准层面，这种 I/O 抽象已经开始协议化：Anthropic 在 2024 年推出 MCP，将其定义为连接 AI 应用与外部数据源、工具、工作流的开放标准，官方甚至把它比作“AI 的 USB-C”；Google 在 2025 年推出 A2A，明确将其定位为与 MCP 互补的 agent-to-agent 开放协议，支持 capability discovery、task lifecycle、长时任务和多模态协同。把这两者合起来看，MCP 很像 **模型到系统的 I/O 总线**，A2A 很像 **agent 到 agent 的互联协议**。citeturn7search0turn7search1turn7search2turn7search3turn10view0turn10view1turn10view2

第五条是 **多智能体作为分布式系统或“互联网中的代理网络”**。AutoGen 把多 agent 会话作为通用基础设施；MetaGPT 用 assembly line 和 SOP 将多 agent 协作显式流程化；Internet of Agents 进一步从“互联网”而不是单机系统来想象 agent 生态，强调 agent integration protocol、动态组队和分布式环境；其后续综述把 capability discovery、task orchestration、communication protocols、consensus/conflict resolution 统合为一个 agent-native 网络基础设施问题。你的“多智能体系统像多核/分布式”并不是牵强附会，而是已经在研究对象层面浮现出来，只是大多数论文还停留在“框架和协议”层，而未与单体 agent 的内存、内核、I/O、权限一起收拢成统一的计算机体系结构。citeturn4search1turn4search2turn24search1turn23view0

最后还要补一条 **认知架构与治理架构**。CoALA 并不从传统计算机体系结构出发，而是从 cognitive architecture 出发，把语言 agent 组织为 modular memory、structured action space 和 generalized decision-making process。这条线对于你的论文非常重要，因为它提供了“为什么这个系统不是纯软件栈拼装，而是真正的 cognition architecture”的理论支点。另一边，ArbiterOS 则指出问题不只是系统抽象不够，更是因为我们在用 deterministic software engineering mental models 去指挥 inherently probabilistic processors；它由此提出 governor、formal instruction set、HAL 这样的治理层概念。把 CoALA 和 ArbiterOS 放进你的框架里，你的文章就不再只是“系统类比”，而会同时拥有认知层与治理层。citeturn19view0turn25view0turn1search6

## 现有工作的共同边界与真正空缺

现有工作最大的共同特点，是**分层都在长，但全栈还没有合拢**。OS 线主要解决 runtime、resource management、isolation、desktop/mobile integration；memory 线主要解决 long context、persistent memory、memory lifecycle；tool/protocol 线主要解决 tool calling、external connectivity、agent interoperability；multi-agent 线主要解决 task decomposition、communication、协同拓扑；cognitive 线主要解决 memory/action/decision 的抽象；security/governance 线主要解决 taint-aware memory、semantic firewall、access control、formal policy enforcement。它们共同逼近你设想的“大模型原生智能计算机”，但没有哪篇主流工作同时把 **概率计算核心、语义记忆层级、工具与外设总线、agent OS/kernel、互联协议、治理安全层、编程模型/指令语义、评价指标** 全部放在同一套体系结构里。这个“没有一篇统一总图”不是空洞判断，而是这些论文各自摘要与作用域直接显示出来的事实。citeturn15view0turn17view0turn18view0turn19view0turn20view0turn21view0turn22view0turn10view0turn10view1turn24search1

你原始想法里最值得保留、但也最需要修正的地方，是“KV cache = CPU cache”这一句。官方开发文档把 KV cache 定义为推理阶段缓存过去 token 的 key/value 或激活状态，以避免重复计算、加快自回归生成；因此它更像**会话级、推理期、微架构内部的状态复用机制**，而不等于传统意义上的完整 cache hierarchy。更稳健的写法，是把 KV cache 看成推理引擎的局部高速状态，把**受管上下文窗口**看成当前 working set，把**外部检索/长期记忆/参数记忆**看成更广义的语义内存层级。Mi 等自己就明确指出：在现有 agent 中，context window 类似主存、数据库类似磁盘，而“cache 模块”反而缺位；MemGPT 与 AgentOS 也都把重点放在 virtual context 和 addressable semantic space，而不是把全部记忆机制等同于原始 KV cache。这样写，你的论文会比单纯照搬类比更严谨。citeturn26search6turn26search20turn14view0turn17view0turn21view2

另一个真正的空缺，是 **层次边界的冲突尚未统一**。如果 LLM 是 OS kernel，那么谁是 CPU？如果 LLM 是 CPU，那么谁来当 deterministic kernel？如果 LLM 是 reasoning kernel，那么真正的可验证控制逻辑又放在哪？Ge 等与 AIOS 把 LLM 放到 OS 核心位置；ArbiterOS 则将其下沉为被 kernel 治理的“Probabilistic CPU”；AgentOS 把它定义成受 structured operating system logic 约束的 reasoning kernel。你论文若能把这组冲突改写成“两平面或三平面架构”，比如“概率执行平面 + 确定性控制平面 + 语义记忆平面”，就会明显高于“简单映射表”式的综述。citeturn16view0turn15view0turn25view0turn21view2

## 为什么你的论文仍然可以更上一个层次

真正更高一级的写法，不应再是“某某像某某”的类比罗列，而应是提出一种 **LLM-native intelligent computer architecture** 或 **agentic computer architecture** 的统一命题。这个命题不是去争“模型到底像 CPU 还是像 OS”，而是说明：传统计算机围绕 deterministic processor、memory hierarchy、I/O、OS、network 和 programming model 组织；未来的智能计算则应围绕 probabilistic model core、semantic memory hierarchy、tool-mediated I/O、agent runtime/kernel、agent interconnect、natural-language-to-formal programming interface、以及 governance/security plane 重新定义层次边界。现有文献已经给了每一层的局部原型，但还缺一个把它们组织成规范性架构的总论。citeturn20view0turn14view0turn17view0turn18view0turn10view0turn10view1turn25view0

这也是为什么你不应把 Codex、Claude Code、OpenClaw 直接写成“操作系统本身”。从更成熟的系统抽象看，它们更像**建立在 runtime、tools、sandboxes、handoffs 和 protocol 之上的 agent applications / shells / user-facing runtimes**。OpenAI 的官方 Agents SDK 文档已经把 orchestration、tool execution、state、guardrails、MCP 和 sandbox agents 明确作为一组系统级能力来呈现；Anthropic 的 MCP 与 Google 的 A2A 又分别把 model-to-system 与 agent-to-agent 交互协议化。这意味着产业工程实践确实正在向“系统软件化”迈进，但**前台产品 ≠ 内核层抽象**。把这个层次关系写清楚，你的论文会从“产品印象总结”上升成“架构边界定义”。citeturn10view3turn10view0turn10view1turn10view2

更进一步，你还可以把“为什么需要新体系结构”写得比现有综述更尖锐。ArbiterOS 已明确指出，agent 工程的危机来自用确定性软件思维去驱动概率型处理器；Aura 又说明，一旦 agent 真正进入操作系统和移动端环境，权限、身份、语义输入污染、记忆 taint 传播都会变成内核级问题。也就是说，这不只是“模型如何更聪明”的问题，而是**当模型变成计算栈中的执行核心以后，传统系统设计中的调度、隔离、最小权限、异常处理、审计、互操作、生命周期管理，必须被重新发明为智能原生版本**。这个表述本身，就已经比“把大模型类比成 CPU”高了一个层次。citeturn25view0turn22view0

## 原创性边界与可发表性判断

如果你现在就以“综述 + 分析 + 架构畅想”的 arXiv 论文形式来写，这个方向**完全可行**，但原创性边界必须画得很准。最不该做的，是宣称“第一次把大模型和计算机体系结构类比起来”——因为 Ge 等、L2MAC、Mi 等已经覆盖了 OS、stored-program computer 和 von Neumann inspired survey 这几条最关键路线，而 MemGPT 与 MemOS 也已经把记忆子系统高度系统化。你真正可以主张的是：**第一次把这些互相独立、甚至互相冲突的类比整合为一套全栈 LLM-native intelligent computing architecture**。citeturn16view0turn20view0turn14view0turn17view0turn18view0

从“贴合你想法的程度”来说，学术上最接近的先验作品并不是那篇 aiXiv 的 *Treating LLM as the CPU*，而是两组更扎实的工作：一组是 Ge 等和 AIOS 代表的 **LLM-as-OS / kernel**；另一组是 L2MAC 与 Mi 等代表的 **von Neumann / automatic computer / computer systems perspective**。前者证明你的直觉已经被系统化过，后者证明“用计算机体系结构重写 agent 栈”已经成为可以成文的研究主题。ArbiterOS、AgentOS、UFO2、Aura 等 2025—2026 的后续工作则说明这个议题还在快速外扩到治理、安全、桌面和移动操作系统层面。正因为如此，你的论文最好的站位，不是再加入一篇平铺直叙的“相似性综述”，而是写成一篇**总架构论文**：它要回答的不是“像不像”，而是“下一代智能计算机应该如何分层、这些层之间的边界在哪里、什么评价标准可以判断这套架构是否成立”。citeturn15view0turn16view0turn20view0turn14view0turn25view0turn21view2turn21view0turn22view0
